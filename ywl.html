USING LEX WITH YACC

Integrating LEX with YACC

In the previous documents, we have noted that YACC is used to generate a parser (YACC documentation) and LEX is used to generate a lexical analayzer (LEX documentation). YACC generates the definition for yyparse() in y.tab.c and LEX generates the definition for yylex() in lex.yy.c. We have also noted that yyparse() repetitively calls yylex() to read tokens from the input stream. Till now, for simplicity we had written a user-defined yylex() in the YACC program. In this section of the document we will use LEX to generate the definition of yylex() and make YACC use this definition for retrieving tokens.

For yyparse() to use yylex(), the definition of yylex() must be available in y.tab.c file. We will add this definiton of yylex() to the Auxiliary functions section of the YACC program. As the definition of yylex() has been generated in the the lex.yy.c file, the lex.yy.c file must be included in the auxiliary functions section of the YACC program. (This results in all of the contents of lex.yy.c including the yylex() definition being copied to y.tab.c) 

		/* Declarations section */
	%%
		/* Rules */
	%%
	#include "lex.yy.c"

		/* Auxiliary Functions */

NOTE: In this case, we must not provide a main() definition in the LEX program calling yylex(), as there already exists a main() function in the YACC program which calls yyparse() which inturn calls yylex().

Recall that yyparse() attempts to parse the given input by calling yylex() to obtain tokens. In the infix to postfix conversion example in the YACC documentation, we had used a user defined definition of yylex() in the YACC program. In this example, the YACC program consists of the declaration of token DIGIT in the declarations section and definition of the token DIGIT in the auxiliary functions section under yylex(). Now we are trying to use the LEX generated yylex() instead of a user defined yylex().

To do this, first we must write a YACC program to generate yyparse(). The token DIGIT must be declared in the declaration section to be used in the Rules section.

The following example shows the declaration of the token DIGIT in the YACC program. For a complete working example click here

		/* YACC Declarations */
	
	%token DIGIT

	%%
		/* Rules */
	%%
		/* YACC Auxiliary Functions */

In the above example we can see that the YACC program contains the declaration of the token DIGIT. We are yet to write a LEX program (defining token DIGIT) to generate yylex(). But before that, there must be some way by which the YACC program can tell the LEX program that the token DIGIT has been declared in the YACC program. This communication is facilitated by the file "y.tab.h" which contains the declarations of all the tokens in the YACC program. The "y.tab.h" is automatically generated by YACC when the """yacc""" command is executed with the -d flag.

user@localhost:~$ yacc -d in2post.y

An example of the contents of y.tab.h file is shown below.

#define DIGIT 253

Note that '253' is a YACC generated constant to represent DIGIT.

The next example example shows the definition of the token DIGIT in the LEX program. For a complete working example click here

		/* LEX Declarations */

	%%
	
	[0-9]+     return DIGIT;
	
	%%
		/* LEX Auxiliary Functions */

The y.tab.h file must be included in the declarations section of the LEX program. As a result of which, LEX has access to the declarations in the YACC program file. No explicit declaration of the token DIGIT is requied in the LEX program as y.tab.h (which contains the declaration of DIGIT) has been included in the declarations section

Inorder to generate lex.yy.c, do:

user@localhost:~$ lex in2post.l

Once y.tab.c and lex.yy.c files have been generated by YACC and LEX respectively, they can be compiled by including lex.yy.c in y.tab.c file (as shown above). The compilation steps and sample input/output of the working example are shown below:

user@localhost:~$ gcc y.tab.c
user@localhost:~$ ./a.out
1+2*3
1 2 3 * +
user@localhost:~$
